# Sub2API 账号聚合页面设计

## 目标

在 UpstreamOps 中新增独立的 Sub2API 聚合页面。页面使用唯一一个已配置目标的 Admin API Key，实时读取该 Sub2API 的全部账号、分组和代理信息，并关联 UpstreamOps 本地托管映射，形成统一的账号与分组总览。

页面允许管理员直接启用或禁用账号调度，只修改远端账号的 `schedulable` 字段，不修改 `active` / `inactive` 状态，不修改 Sub2API 源码。

## 已确认范围

- 系统只会配置一个目标 Sub2API，不设计多目标选择或跨目标聚合。
- 聚合范围包含目标 Sub2API 中的全部账号，包括 UpstreamOps 托管账号和非托管账号。
- 页面同时展示账号明细和分组汇总。
- 页面提供单账号调度开关。
- 不在本页面创建、删除、编辑账号，也不修改账号启用状态、凭据、分组绑定或代理配置。
- 不增加定时缓存；每次进入页面或手动刷新时读取远端最新状态。
- 本期不展示请求量、失败率、TTFT、TPS 等用量指标。这些指标不在现有账号、分组接口响应中。

## 方案选择

采用“后端实时聚合、前端一次加载”的方式。

后端负责读取已配置目标、解密 Admin API Key、请求远端账号/分组/代理接口，并关联本地托管映射。前端只接收页面需要的脱敏 DTO，不接触 Admin API Key 或账号凭据。

不采用前端分别请求多个数据源并自行拼装。该方式会把远端响应兼容、部分失败处理和托管识别逻辑散落到 UI 中。不采用数据库定时缓存，避免页面状态滞后以及调度更新后的缓存一致性问题。

## 页面与导航

新增独立路由：

```text
/sub2api-overview
```

顶部导航增加带图标的“Sub2API 聚合”入口，移动端保留可访问性和工具提示。页面沿用现有 `AppShell` 的最大宽度、间距、颜色和组件规范，不新增另一套视觉系统。

页面自上而下分为四部分：

1. 标题与刷新操作。
2. 账号统计摘要。
3. 分组汇总表。
4. 账号筛选栏和账号明细表。

### 账号统计摘要

展示以下互斥或可验证指标：

- 账号总数。
- 状态为启用的账号数。
- 当前可调度账号数。
- UpstreamOps 托管账号数。
- 非托管账号数。

“启用账号”取远端账号 `status`；“可调度账号”取 `schedulable`，两者不能混为同一个状态。

### 分组汇总

每个远端分组显示：

- 分组名称和平台。
- 分组状态。
- 计费倍率。
- 绑定账号数。
- 其中启用账号数。
- 其中可调度账号数。

同一账号可以绑定多个分组，因此会计入每个关联分组的统计。没有分组的账号单独计入“未分组”汇总行，但不伪造远端分组 ID。

### 账号明细

每个远端账号只显示一行，所属多个分组以标签形式展示。字段包括：

- 账号名称和远端 ID。
- 平台及账号类型。
- 所属分组。
- 账号状态。
- 调度状态及调度开关。
- 账号倍率。
- 并发数和优先级。
- 代理名称。
- 托管状态；托管账号同时显示对应同步分组名称。

支持按账号名称搜索，并按平台、分组、账号状态、调度状态和托管状态筛选。筛选在当前完整数据集上执行，不向远端重复发送筛选请求。

## 后端接口

### 聚合查询

新增：

```text
GET /api/upstream-sync/overview
```

服务层按以下顺序处理：

1. 读取已配置目标；没有目标时返回可识别的未配置错误。
2. 若存在多个目标，拒绝隐式选择并返回配置冲突错误，避免操作错误站点。
3. 解密目标 Admin API Key，仅在服务端内存中构造 `AdminTarget`。
4. 请求远端全部账号、全部分组和全部代理。
5. 读取本地同步分组与托管账号映射。
6. 按远端账号 ID 关联托管来源，按分组 ID 和代理 ID补全显示名称。
7. 计算摘要与分组汇总，返回脱敏 DTO。

账号列表必须完整分页拉取，不能依赖现有固定 `page_size=1000`。分页在返回空页或不足一页时结束，并设置合理页数上限防止异常远端无限响应。

聚合响应只包含页面展示字段。明确禁止返回：

- 目标 Admin API Key。
- 远端账号 `credentials`。
- 代理用户名、密码或完整代理连接串。
- 源渠道 API Key、Token 或密码。

远端任一核心接口失败时，本次聚合查询失败并返回明确错误，不返回混合时间点的半份数据。前端保留上一次成功结果并显示错误提示。

### 调度更新

新增：

```text
PUT /api/upstream-sync/accounts/:account_id/schedulable
```

请求体：

```json
{
  "schedulable": true
}
```

服务端验证账号 ID 为正整数，读取唯一目标并使用已有 AdminClient 更新 `schedulable`。响应返回更新后的账号 ID 和调度状态，不返回完整远端账号对象。

该接口不修改 `status`。远端返回错误时，前端恢复开关原状态并显示错误信息。同一账号请求进行期间禁用其开关，避免重复提交和响应乱序。

## 托管账号识别

托管判断以 `upstream_sync_managed_accounts` 的映射记录为准，不依赖账号名称前缀猜测。

映射通过 `sync_group_id` 关联同步分组，再验证同步分组属于当前目标。只有目标一致且 `target_account_id` 与远端账号 ID 相同，才标记为托管账号。映射记录存在但远端账号不存在时不计入远端账号总数；该清理仍由现有同步流程负责。

## 状态与错误处理

- 未配置目标：显示空状态和前往“系统设置 → 上游动态同步”的入口。
- 目标被禁用：允许查看和刷新，但在页面中明确标识目标已禁用；手动调度操作仍需服务端使用有效配置执行。
- Admin Key 失效或站点不可达：保留上一次成功数据，显示刷新失败消息。
- 分组或代理引用缺失：使用 `分组 #ID` 或 `代理 #ID` 作为非敏感回退文本，不丢弃账号。
- 调度更新成功：只更新对应账号、摘要和分组计数，无需整页清空重载。
- 调度更新失败：恢复原值并显示远端错误。

## 安全边界

- Admin API Key 继续使用现有加密存储和服务端解密链路。
- 浏览器不直接请求目标 Sub2API，避免密钥暴露和跨域问题。
- 日志不得记录 Admin Key、账号凭据或完整代理认证信息。
- API DTO 使用显式字段，不直接序列化 `sub2api.AdminAccount`。
- 调度接口只接受布尔值，不接受任意远端账号更新负载。
- 不修改 Sub2API 源码、数据库和部署配置。

## 测试范围

### 后端

- 唯一目标下正确聚合账号、分组、代理和托管映射。
- 没有目标和多个目标时返回明确错误。
- 多页账号能够完整拉取并终止分页。
- 多分组账号分别计入对应分组统计。
- 未分组账号计入“未分组”汇总。
- 托管识别只使用当前目标的有效映射。
- 响应中不包含 `credentials`、Admin Key 或代理认证字段。
- 调度接口只修改 `schedulable`，不修改 `status`。
- 非法账号 ID、非法请求体和远端失败正确返回错误。

### 前端

- 加载、空数据、失败和成功状态正确展示。
- 摘要、分组统计和账号表字段正确。
- 搜索及平台、分组、状态、调度、托管筛选正确组合。
- 调度开关请求中禁用，成功后更新计数，失败后恢复原值。
- 桌面和移动端无文字溢出或控件重叠。

### 回归与安全扫描

- 运行 `go test ./...`。
- 运行前端 ESLint、TypeScript 检查和生产构建。
- 使用桌面和移动端视口检查页面截图与交互。
- 执行常见硬编码密钥、Token、密码和私钥特征扫描。
